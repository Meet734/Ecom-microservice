SECTION 2 — PROJECT OVERVIEW (CONTEXT FOR THE AGENT)
────────────────────────────────────────────────────────────────────────────────

Project name    : ShopStack
Architecture    : Microservices (8 services + infrastructure)
Language        : Node.js with ES Modules (import/export) — NOT CommonJS
Runtime         : Node.js 20 (inside Docker containers, node:20-alpine)
Database        : PostgreSQL 16 — one instance per service
ORM             : Sequelize
Auth model      : JWT — access token (15min) + refresh token (7d)
Token storage   : Redis (blacklist), PostgreSQL (refreshToken column)
Message broker  : RabbitMQ — topic exchange named "shopstack.events"
Search          : Elasticsearch 8.11 — index "products"
Logging         : Structured JSON logs — Elasticsearch index "application_logs"
Log shipping    : Fluent Bit (reads Docker container logs, ships to ES)
Visualization   : Kibana (connects to Elasticsearch)
Cache           : Redis 7.2
API Gateway     : Nginx (port 3000)
Containerized   : Full Docker Compose stack

Services and ports:
  api-gateway          → 3000  (Nginx)
  auth-service         → 3001
  user-service         → 3002
  product-service      → 3003
  order-service        → 3004
  payment-service      → 3005
  inventory-service    → 3006
  notification-service → 3007

Infrastructure ports:
  RabbitMQ AMQP        → 5672
  RabbitMQ Mgmt UI     → 15672
  Redis                → 6379
  Elasticsearch        → 9200
  Kibana               → 5601

SECTION 5 — DATABASE SCHEMA CHECKS
────────────────────────────────────────────────────────────────────────────────

PURPOSE: Verify each service's database schema matches the agreed design.
Sequelize sync creates tables on startup — verify actual tables in each DB.

Commands to inspect schemas:
  docker exec -it ecom_auth_db psql -U auth_user -d auth_db -c "\dt"
  docker exec -it ecom_auth_db psql -U auth_user -d auth_db -c "\d users"

Repeat for each database.

--- 5.1 AUTH SERVICE — auth_db ---

Table: users
Expected columns:
  id            UUID, primary key, not null
  email         VARCHAR, unique, not null
  password      VARCHAR, not null
  role          ENUM or VARCHAR — values: customer, seller, admin
  isVerified    BOOLEAN, default false
  refreshToken  TEXT, nullable
  createdAt     TIMESTAMP, not null
  updatedAt     TIMESTAMP, not null

Checks:
  - email column has UNIQUE constraint
  - password column stores bcrypt hash (starts with "$2b$12$")
    Verify: SELECT password FROM users LIMIT 1;
    Expected: value starts with "$2b$" not plain text
  - refreshToken is nullable (users who are logged out have NULL here)
  - id is UUID type not INTEGER

--- 5.2 USER SERVICE — user_db ---

Table: user_profiles
Expected columns:
  id          UUID, primary key
  userId      UUID, not null (references auth user — no FK across services)
  firstName   VARCHAR
  lastName    VARCHAR
  phone       VARCHAR, nullable
  avatarUrl   TEXT, nullable
  createdAt   TIMESTAMP
  updatedAt   TIMESTAMP

Table: addresses
Expected columns:
  id          UUID, primary key
  userId      UUID, not null
  label       VARCHAR (home, work, etc.)
  street      TEXT
  city        VARCHAR
  state       VARCHAR
  pincode     VARCHAR
  country     VARCHAR, default 'India'
  isDefault   BOOLEAN, default false
  createdAt   TIMESTAMP
  updatedAt   TIMESTAMP

--- 5.3 PRODUCT SERVICE — product_db ---

Table: products
Expected columns:
  id          UUID, primary key
  name        VARCHAR, not null
  description TEXT
  price       DECIMAL (not FLOAT — precision matters for money)
  stock       INTEGER, default 0
  category    VARCHAR
  imageUrls   ARRAY or JSONB
  sellerId    UUID
  isActive    BOOLEAN, default true
  createdAt   TIMESTAMP
  updatedAt   TIMESTAMP

Table: categories
Expected columns:
  id          UUID, primary key
  name        VARCHAR, not null
  slug        VARCHAR, unique
  parentId    UUID, nullable (self-referential for subcategories)

Check: price column must be DECIMAL or NUMERIC, never FLOAT.
FLOAT stores money imprecisely — this is a correctness bug if violated.

--- 5.4 ORDER SERVICE — order_db ---

Table: carts
  id        UUID, primary key
  userId    UUID, not null, unique (one cart per user)
  createdAt TIMESTAMP
  updatedAt TIMESTAMP

Table: cart_items
  id              UUID, primary key
  cartId          UUID, not null
  productId       UUID, not null
  quantity        INTEGER, not null, min 1
  priceSnapshot   DECIMAL, not null
  createdAt       TIMESTAMP
  updatedAt       TIMESTAMP

CRITICAL CHECK — priceSnapshot:
  This column must exist and must NOT be nullable.
  It captures the product price at time of adding to cart.
  If this column is missing or nullable, the order service has a
  fundamental design flaw — product price changes would affect
  historical orders.

Table: orders
  id              UUID, primary key
  userId          UUID, not null
  items           JSONB, not null (snapshot of ordered items)
  totalAmount     DECIMAL, not null
  status          ENUM: pending, confirmed, shipped, delivered, cancelled
  shippingAddress JSONB, not null
  paymentId       UUID, nullable (filled after payment)
  createdAt       TIMESTAMP
  updatedAt       TIMESTAMP

--- 5.5 PAYMENT SERVICE — payment_db ---

Table: payments
  id                      UUID, primary key
  orderId                 UUID, not null
  userId                  UUID, not null
  amount                  DECIMAL, not null
  currency                VARCHAR, default 'inr'
  status                  ENUM: pending, success, failed, refunded
  stripePaymentIntentId   VARCHAR, nullable
  stripeSessionId         VARCHAR, nullable
  createdAt               TIMESTAMP
  updatedAt               TIMESTAMP

--- 5.6 INVENTORY SERVICE — inventory_db ---

Table: inventories
  id                  UUID, primary key
  productId           UUID, not null, unique
  totalStock          INTEGER, not null, default 0
  reservedStock       INTEGER, not null, default 0
  lowStockThreshold   INTEGER, default 10
  createdAt           TIMESTAMP
  updatedAt           TIMESTAMP

CRITICAL CHECK — availableStock:
  This should be a VIRTUAL field computed as (totalStock - reservedStock)
  NOT a stored column. If it is stored, any update to totalStock or
  reservedStock without updating availableStock creates inconsistency.

Table: inventory_transactions
  id            UUID, primary key
  productId     UUID, not null
  type          ENUM: STOCK_IN, STOCK_DEDUCT, RESERVE, RELEASE
  quantity      INTEGER, not null
  referenceId   UUID, nullable (orderId or paymentId)
  note          TEXT, nullable
  createdAt     TIMESTAMP

--- 5.7 NOTIFICATION SERVICE — notification_db ---

Table: notification_logs
  id              UUID, primary key
  event_id        UUID, not null, UNIQUE  ← CRITICAL for idempotency
  user_id         UUID, not null
  type            ENUM or VARCHAR
  channel         ENUM: email, sms, push
  recipient       VARCHAR, not null
  status          ENUM: pending, sent, failed, skipped
  retry_count     INTEGER, default 0
  error_details   TEXT, nullable
  metadata        JSONB, nullable
  sent_at         TIMESTAMP, nullable
  createdAt       TIMESTAMP
  updatedAt       TIMESTAMP

CRITICAL CHECK — event_id UNIQUE constraint:
  Run: \d notification_logs
  Confirm UNIQUE constraint exists on event_id column.
  Without this, idempotency fails at the database level.

Table: user_notification_preferences
  id              UUID, primary key
  user_id         UUID, not null, UNIQUE
  email_enabled   BOOLEAN, default true
  sms_enabled     BOOLEAN, default false
  order_updates   BOOLEAN, default true
  promotions      BOOLEAN, default true
  createdAt       TIMESTAMP
  updatedAt       TIMESTAMP

CHECKLIST 5:
  [PARTIAL] users table has correct columns and UUID primary key (uses password_hash, is_verified, and separate refresh_tokens table)
  [PASS]    passwords stored as bcrypt hash not plain text
  [PASS]    email column has UNIQUE constraint
  [PARTIAL] user_profiles and addresses tables exist with correct columns (tables profiles and addresses with snake_case fields and profile_id instead of userId, line1/2 instead of street)
  [FAIL]    products table uses DECIMAL not FLOAT for price (uses INTEGER representing paise/cents)
